# taka-skills 项目记忆

## 项目定位

`taka-skills` 是一个 skill 整合仓库，用来沉淀可复用的 skill，并让这些 skill 能够适配多个 Agent 平台，包括 `Claude Code`、`Codex`、`OpenClaw`，以及其他类似的 AI 编码工具或 Agent 运行环境。

这个仓库应该保持“理解不同平台”，但不要被某一个平台强绑定。优先编写能被多个 Agent 复用的说明、资源、脚本和示例；如果某个平台有特殊行为，把差异写在清晰的兼容说明里。

## 仓库结构

- 每个 skill 应该放在独立目录中。
- 每个 skill 目录都应该能被单独理解：用途、触发场景、所需文件、平台兼容注意事项都应该容易找到。
- 共享约定应放在仓库级文档里，不要在每个 skill 中重复维护。
- 不要把互不相关的 skill 混在同一个目录里。如果一个 skill 扩展成多个独立能力，应拆分成多个 skill 目录。

## Skill 编写原则

- 说明要精确、可执行，方便 Agent 直接遵循。
- 当不同平台行为不一致时，明确写出平台名称，例如 `Claude Code`、`Codex`、`OpenClaw`。
- 优先使用可迁移的文件格式和简单的目录结构。
- 示例只在能帮助理解执行方式或集成方式时添加。
- 不要假设不同 Agent 运行环境拥有相同的工具、钩子、记忆模型或插件格式。

## Codex 工作约定

- 编辑某个 skill 前，先查看它的目录结构和附近文档。
- 保持现有 skill 边界，不要在修改一个 skill 时顺手重构无关目录。
- 改动要聚焦，方便审阅和回滚。
- 将 `AGENTS.md` 作为 Codex 兼容工作流的项目级记忆入口。
- 如果要支持新的 Agent 平台，应先记录兼容预期，而不是静默改变已有行为。

## 文档约定

- `README.md` 面向人类读者，用来介绍仓库概览。
- `AGENTS.md` 面向 Agent，用来记录项目级记忆和工作约定。
- 如果未来某个平台需要独立配置文件，应与本文件保持一致，并引用这里记录的项目意图。

## Skill 生成流程约束

- 生成或更新 skill 前，先明确真实使用场景：用户会怎样请求、输入是什么、期望产物是什么、Agent 容易在哪些地方犯错。
- 优先从真实任务、项目文档、运行手册、代码审查意见、历史修复记录中提炼 skill，不要只基于泛泛的“最佳实践”生成空洞流程。
- 创建新 skill 前，先检查仓库里是否已有相近 skill；能扩展已有 skill 的，不要创建重复能力。
- 生成前应先给出简短设计：skill 名称、触发条件、非适用场景、核心流程、目录结构、验证方式。
- 对复杂 skill，应至少准备 2-3 个真实 eval 用例；包含普通场景、边界场景，以及一个容易误触发或不应触发的场景。
- 更新已有 skill 时，先说明要解决的问题，再修改最小必要内容，避免顺手重写整个 skill。

## Skill 目录结构约束

- 每个 skill 使用独立目录，目录名必须与 `SKILL.md` frontmatter 中的 `name` 一致。
- skill 名称使用小写字母、数字和连字符，长度不超过 64 个字符；不要以下划线、大写字母、空格、连续连字符、开头连字符或结尾连字符命名。
- 每个 skill 必须包含 `SKILL.md`。可选目录按用途使用：
  - `scripts/`：可执行脚本、校验器、转换器、重复性工具。
  - `references/`：按需读取的详细参考、API 文档、领域规则、长示例。
  - `assets/`：模板、图片、样例文件、字体、配置片段等输出资源。
  - `evals/`：用于验证 skill 触发和产物质量的测试用例。
- 不要在 skill 目录里堆放无关文档，例如独立的 `README.md`、安装笔记、变更日志或创建过程记录；共享说明放到仓库级文档，必要说明写进 `SKILL.md`。
- `SKILL.md` 引用其他文件时使用相对路径，并尽量保持一层引用，不要形成多层跳转链。

## SKILL.md 元数据约束

- `SKILL.md` 必须以 YAML frontmatter 开始，至少包含 `name` 和 `description`。
- `description` 是 Agent 判断是否加载 skill 的主要入口，应描述“何时使用”和“解决什么任务”，不要只写“帮助处理某某问题”。
- `description` 应包含用户可能使用的关键词、同义词和相关场景，但不能过宽；应同时考虑应触发和不应触发的近似场景。
- 本仓库面向中文用户，正文以中文为主；`description` 可使用中文、英文或中英混合，以触发准确性和跨平台兼容性为优先。
- 只有在确实存在环境要求时才添加 `compatibility`，例如目标平台、系统依赖、网络权限、Python/Node 版本。
- `allowed-tools` 等实验性字段必须注明平台支持差异；不要假设所有 Agent 平台都会识别。

## SKILL.md 正文约束

- 正文只放 Agent 执行任务所需的信息，避免解释 Agent 已经知道的通用概念。
- 推荐结构：概览、何时使用、何时不要使用、核心流程、输入输出约定、常见坑、验证方式、可用脚本或参考文件。
- 流程要可执行：写清步骤、判断条件、默认选择和失败后的修正方式。
- 当多个方案都可行时，给出默认方案，再简短说明替代方案；不要把一长串选项交给 Agent 自己猜。
- 对脆弱、危险或必须一致的操作，要给出严格步骤；对开放性任务，可以给原则和判断标准。
- 常见坑应写得具体，记录 Agent 如果不知道就容易犯错的项目事实、平台差异、命名差异或边界条件。
- 长参考、API 细节、平台差异矩阵、完整示例应放入 `references/`，并在 `SKILL.md` 中说明何时读取。
- 控制上下文体量：`SKILL.md` 建议保持在 500 行以内，核心说明优先，避免把整份参考资料塞进主文件。
- 不要在 `description` 或正文中写会绕过用户意图、系统规则、权限确认或安全边界的指令。

## 脚本与资源约束

- 只有当逻辑需要稳定复现、容易手写出错、或会被反复执行时，才放入 `scripts/`。
- 脚本应非交互式运行，不要等待 TTY 输入、密码框或确认菜单；所有输入通过参数、环境变量或 stdin 提供。
- 脚本应支持简洁的 `--help`，说明参数、默认值、示例和退出码。
- 脚本错误信息要可修正：说明收到什么、期望什么、下一步该怎么做。
- 输出优先使用 JSON、CSV、TSV 等结构化格式；结构化结果写 stdout，进度、警告、诊断写 stderr。
- 有状态、批量或破坏性脚本应支持 `--dry-run`、幂等执行和显式确认参数。
- 引入依赖时固定版本或写清版本范围；不要依赖隐式全局环境。
- `assets/` 中的模板和样例应服务于 skill 输出，不要存放与执行无关的素材。

## 验证与质量约束

- 非平凡 skill 应提供 eval：真实 prompt、期望输出、必要输入文件，以及可检查的成功标准。
- 触发验证要同时覆盖 should-trigger 和 should-not-trigger 场景，避免 skill 过窄或过宽。
- 产物验证要有证据：文件是否生成、格式是否有效、字段是否齐全、断言是否通过，不要只写“看起来不错”。
- 修改 description 后，应重新检查触发场景，确认没有因为新增关键词导致误触发。
- 复杂 skill 建议与无 skill 或旧版本 skill 做基线对比，确认它确实提升质量，而不是只增加上下文成本。
- 如果验证依赖特定平台能力，应在 `compatibility` 或正文中写明。

## 安全与兼容约束

- skill 不应静默修改自身、其他 skill、Agent 全局配置或用户环境；确需修改时必须让用户明确知道改动范围。
- 不要在 skill 中保存密钥、令牌、个人隐私数据或不可公开的内部信息；需要凭据时只说明读取方式和权限边界。
- 对网络访问、文件删除、批量改写、发布、部署、支付等高风险操作，必须写清前置确认、dry-run 或人工审批点。
- 不要把平台私有能力伪装成通用能力。若某流程只适用于 `Claude Code`、`Codex` 或 `OpenClaw`，必须明确标注。
- 引入第三方脚本、模板或外部说明时，应记录来源、许可和信任假设，避免 skill 成为供应链风险入口。
